# 完成前验证

## 概览

没有验证证据就宣称“完成了”，这不是高效，而是不诚实。

**核心原则：**任何结论都先拿证据，再下判断。

**违反这条规则的字面要求，也是在违反它的精神。**

## 铁律

```text
没有新的验证证据，就不能宣称工作已完成
```

如果你没有在当前消息对应的动作里实际运行验证命令，就不能说“通过了”“修好了”“完成了”。

如果当前是“代码刚写完”的状态，也不能直接以 `git commit` 代替收尾；新需求还必须补齐 `/doc/feat/feat_xxxx.md` 后，才有资格宣称完成。

## 验证闸门

```text
在宣称任何状态、表达任何满意或成功之前：

1. IDENTIFY：找出哪条命令能证明这个结论
2. RUN：完整执行这条命令，而且要是新的、完整的执行
3. READ：通读输出，检查退出码，确认失败数量
4. VERIFY：输出是否真的支持你的结论？
   - 如果不支持：按真实状态汇报，并附上证据
   - 如果支持：带着证据陈述结论
5. DOCUMENT：如果是新需求，检查 `/doc/feat/feat_xxxx.md` 是否已更新，SQL、参数、接口、资源、版本依赖等改动是否记录完整
6. ONLY THEN：只有这时才能宣称完成
```

少做任何一步，都不叫验证。

## 常见错误对照

| 你想宣称的内容 | 真正需要的证据 | 什么不算证据 |
|----------------|----------------|--------------|
| 测试通过 | 测试命令输出显示 0 failure | 之前跑过一次、主观觉得应该通过 |
| Linter 干净 | linter 输出 0 error | 只检查了部分文件 |
| 构建成功 | build 命令退出码为 0 | linter 通过、日志“看起来还行” |
| bug 已修复 | 复现原问题的验证现在通过 | 代码改了、你觉得应该修好了 |
| 回归测试有效 | 走完 red-green 验证闭环 | 测试只通过过一次 |
| 新需求交付已完整 | `/doc/feat/feat_xxxx.md` 已更新且内容齐全 | 口头说“后面再补版本说明” |
| 子代理完成任务 | 代码 diff 和验证结果支持它的说法 | 子代理口头说“搞定了” |
| 需求已实现 | 逐条对照清单核验 | “测试都过了” |

## 红旗信号

出现以下任何一种，就应立即停下：

- 准备说 “should”“probably”“seems”
- 在验证前先表达满意，例如 “Great”“Perfect”“Done”
- 没跑验证就想提交、推送、开 PR
- 代码写完后想直接 `git commit`
- 直接信任代理或脚本的“成功”汇报
- 只做部分验证就想外推出整体结论
- 想着“就这一次”
- 已经疲惫，只想尽快结束
- **任何暗示成功或完成的话术，但你还没验证**

## 防止自我合理化

| 借口 | 现实 |
|------|------|
| “现在应该可以了” | 那就去跑验证 |
| “我很有把握” | 把握不是证据 |
| “就这一次” | 没有例外 |
| “Linter 过了” | Linter 不等于构建成功 |
| “代理说成功了” | 你自己验证后再说 |
| “我太累了” | 疲劳不是豁免理由 |
| “做点局部检查就够了” | 局部验证不能证明整体 |
| “我换个说法就不算宣称成功” | 规则看实质，不看措辞 |

## 关键模式

**测试：**

```text
正确：跑测试 -> 看到 34/34 通过 -> 再说“测试全部通过”
错误：说“现在应该过了” / “看起来没问题”
```

**回归测试（TDD Red-Green）：**

```text
正确：写测试 -> 跑通 -> 回退修复 -> 确认测试必须失败 -> 恢复修复 -> 再跑通
错误：只写了一个回归测试，就说“已经验证了”
```

**构建：**

```text
正确：实际运行 build，并看到退出码为 0 -> 再说“构建通过”
错误：只因为 linter 过了，就说“构建没问题”
```

**需求实现：**

```text
正确：重新读计划 -> 逐条列检查项 -> 一项项核对 -> 再报告完成或缺口
错误：测试过了，就说“这一阶段完成”
```

**代理委托：**

```text
正确：代理说完成 -> 你自己检查 diff -> 自己跑验证 -> 再报告真实状态
错误：直接相信代理汇报
```

## 为什么这件事重要

来自历史失败案例的共性问题：

- 用户不再相信“已经完成”的说法
- 未定义函数或缺失分支被带进提交
- 需求未真正完成，却被错误地宣布完成
- 因错误收尾引发返工，浪费时间

## 何时必须使用

**凡是在下面这些动作前，都必须应用本 skill：**

- 任何形式的“成功 / 完成 / 已修复”表述
- 任何满意、肯定、结案式表达
- 提交、开 PR、标记任务完成
- 代码写完后准备做任何 Git 动作
- 切换到下一个任务
- 根据代理结果向外汇报

这条规则对以下情况一律生效：

- 原句
- 近义表达
- 暗示性措辞
- 任何会让人理解为“已经完成 / 正确”的表达

## 底线

不要走捷径。

先跑命令，读懂输出，再给结论。

这是强制要求。
