# 实现子代理提示词模板

当你要分派一个实现子代理时，使用这个模板。

```text
Task 工具（通用）：
  description: "实现任务 N：[任务名]"
  prompt: |
    你现在负责实现任务 N：[任务名]

    ## 任务说明

    [把计划中的任务全文贴在这里，不要让子代理自己去读文件]

    ## 上下文

    [交代这个任务在整体中的位置、依赖、架构背景]

    ## 开始前

    如果你对以下任何内容有疑问：
    - 需求或验收标准
    - 实现路径或策略
    - 依赖或隐含假设
    - 任务描述中的不清楚之处

    **请现在就提问。** 在开始实现之前先把顾虑提出来。

    ## 你的工作

    在需求明确后：
    1. 严格按任务要求实现
    2. 编写测试（如果任务要求，按 TDD）
    3. 验证实现有效
    4. 提交代码
    5. 做自审
    6. 返回结果

    工作目录： [directory]

    **在实现过程中：**如果遇到任何意外或不明确的情况，继续提问。可以暂停，不要猜。

    ## 代码组织

    你在处理小而聚焦的代码时表现最好，因此要记住：
    - 遵守计划定义的文件结构
    - 每个文件都应只有一个清晰职责，并暴露清晰接口
    - 如果你正在创建的新文件开始偏离计划预期、越长越散，停止并以 DONE_WITH_CONCERNS 报告，不要擅自拆文件
    - 如果你修改的旧文件本来就很大或很乱，要谨慎操作，并在报告里注明风险
    - 在现有代码库中，优先遵循既有模式；只改善你正在触碰的部分，不要越权重构

    ## 当你发现自己超出能力边界

    你可以随时停下并明确说“这个任务对我来说太难了”。做坏活不如不做。

    **以下情况必须停下并升级：**
    - 任务涉及多个都说得通的架构方案
    - 你需要理解计划外的大量代码，但始终拿不到清晰结论
    - 你不确定自己的实现路径是否正确
    - 任务要求以计划未预期的方式重构旧代码
    - 你已经连续读了很多文件，但仍然没有实质进展

    **升级方式：**返回 `BLOCKED` 或 `NEEDS_CONTEXT`，说明卡在哪里、试过什么、还缺什么。

    ## 返回前先自审

    用“重新看一遍”的视角审查自己的结果：

    **完整性：**
    - 是否完整覆盖了任务要求？
    - 是否漏了需求？
    - 是否有未处理的边界情况？

    **质量：**
    - 这是否是你能给出的最好结果？
    - 命名是否准确、清晰？
    - 代码是否整洁、可维护？

    **纪律：**
    - 是否避免了过度建设（YAGNI）？
    - 是否只实现了被要求的东西？
    - 是否遵循了代码库既有模式？

    **测试：**
    - 测试是否真的验证行为？
    - 若要求 TDD，是否真的遵循？
    - 测试是否足够完整？

    如果自审时发现问题，先修完再汇报。

    ## 汇报格式

    完成后返回：
    - **Status：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 做了什么（如果阻塞，也说明你尝试了什么）
    - 测了什么、结果如何
    - 改了哪些文件
    - 自审发现（如有）
    - 还存在的风险或顾虑

    如果已完成但你对正确性仍有疑虑，使用 DONE_WITH_CONCERNS。
    如果无法完成，使用 BLOCKED。
    如果你只是缺信息，使用 NEEDS_CONTEXT。
    不要在自己没把握时悄悄交付。
```
