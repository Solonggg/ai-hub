# 编写实现计划

## 概览

编写一份完整的实现计划，并假设执行这份计划的工程师对当前代码库几乎没有上下文、审美也未必可靠。你要把他真正需要知道的东西写全：每个任务改哪些文件、写什么代码、查哪些文档、怎么测试、怎么验证。计划必须拆成可以直接执行的小任务。坚持 DRY、YAGNI、TDD，并要求频繁验证；代码写完后不允许直接提交 git。

你可以假设执行者是熟练工程师，但：

- 对当前工具链和业务上下文几乎不了解
- 对测试设计不一定足够敏感

**开始时要明确说明：**“我正在使用 `writing-plans` skill 来编写实现计划。”

**上下文要求：**通常应在由 `brainstorming` 阶段创建好的独立 worktree 中执行，并以同一目录下的 `requirements.md` 和 `design.md` 为输入。

**默认保存路径：**`doc/specs/<需求目录名>/tasks.md`

- 如果目录不存在，先自动创建目录
- `需求目录名` 必须直接沿用 `requirements.md` 所在目录名，不能重新按当前日期或新的功能简称再拼一个目录
- 如果当前上下文已经给出 `requirements.md` 或 `design.md` 路径，直接取其父目录
- 如果当前上下文没有路径，但项目中已经存在本次需求对应的 `requirements.md`，先定位该文件，再沿用其目录

**新需求的附加交付：**

- 计划里必须明确本次需求完成后要补充 `/doc/feat/feat_xxxx.md`
- `xxxx` 优先沿用项目既有版本号命名；如果当前没有明确文件名，计划中必须显式写出“先向用户确认 feat 文件名”，不能跳过
- 如果涉及 SQL、参数、接口、资源、版本依赖等变更，计划必须包含把这些信息同步写入 `/doc/feat/feat_xxxx.md` 的任务

## 范围检查

如果 `requirements.md` 和 `design.md` 同时覆盖了多个彼此独立的子系统，那它本该在 brainstorming 阶段就被拆开。若没有拆，应先建议拆成多份文档与多份计划，每个子系统单独一份。每份计划都必须能独立产出可运行、可测试的软件。

## 文件结构先行

在定义任务之前，先列出本次会新增或修改哪些文件，并说明每个文件负责什么。这一步会锁定你的拆分边界。

- 单个文件应有清晰职责和稳定接口
- 优先小而聚焦的文件，避免大而全
- 经常一起变的内容应该放在一起，按职责拆分，而不是机械按技术层次拆
- 在现有代码库中要尊重既有模式
- 如果某个本次必改文件已经大到难以维护，把拆分纳入计划是合理的

这一步会直接影响后续任务如何拆解。每个任务都应当能形成自洽、可理解的增量。

## 任务粒度

**每一步只做一个动作，控制在 2-5 分钟：**

- “写一个失败测试” -> 一步
- “运行它并确认失败” -> 一步
- “写通过该测试所需的最小实现” -> 一步
- “再跑测试并确认通过” -> 一步
- “整理当前变更或更新 `/doc/feat/feat_xxxx.md`” -> 一步

## 计划文档头部

**每一份计划都必须以下面的头部开场：**

```markdown
# [功能名] 实现计划

> **给代理式执行者：** 必须使用 `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans`，按任务逐项执行本计划。步骤统一使用复选框语法 `- [ ]` 进行追踪。代码写完后先评审、先验证，不要直接 `git commit`。

**目标：** [一句话说明这份计划最终要构建什么]

**架构：** [2-3 句话说明整体方案]

**技术栈：** [关键技术 / 库]

---
```

## 任务结构模板

````markdown
### 任务 N： [组件名]

**文件：**
- 新建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：写失败测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试并确认失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：FAIL，报错为 “function not defined”

- [ ] **步骤 3：写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试并确认通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：PASS

- [ ] **步骤 5：整理结果，不直接提交 git**

```bash
git diff -- tests/path/test.py src/path/file.py
```

如果本次属于新需求，还应补一条步骤：

- [ ] **步骤 6：更新 `/doc/feat/feat_xxxx.md`**

```markdown
补充更新说明，并记录 SQL、参数、接口、资源、版本依赖等相关变更
```
````

## 不允许出现占位内容

每一步都必须提供执行者真正需要的实际内容。下面这些都是**计划失败**：

- `TBD`、`TODO`、“稍后实现”“细节待补”
- “补上适当的错误处理”“加校验”“处理边界情况”
- “为上述内容写测试”（但不写测试代码）
- “参照任务 N”（任务之间不应互相依赖阅读顺序）
- 只说要做什么，但不展示怎么做
- 在任何任务中引用了从未定义过的类型、函数、方法

## 记住

- 永远写精确文件路径
- 每个代码步骤都给出完整代码
- 每条命令都给出精确写法和预期输出
- 始终贯彻 DRY、YAGNI、TDD、频繁验证
- 不要在计划里安排“代码写完立即 git commit”
- 新需求的计划必须包含 `/doc/feat/feat_xxxx.md` 更新任务

## 自审

整份计划写完后，必须自己再审一遍。这不是派子代理，而是你自己做的检查：

### 1. Spec 覆盖

快速过一遍 `requirements.md` 与 `design.md` 的每项要求。你能否明确指出由哪个任务实现？如果有空缺，就补上。

### 2. 占位符扫描

搜索计划中是否出现 “No Placeholders” 章节提到的红旗内容，有就修掉。

### 3. 类型与命名一致性

后续任务中使用的类型、方法签名、属性名，必须与前文定义一致。  
例如任务 3 写了 `clearLayers()`，任务 7 却写成 `clearFullLayers()`，这就是计划 bug。

如果发现问题，就直接在计划内修正；不需要额外再走一轮审查。

## 执行交接

计划保存后，给用户两个执行选项：

```text
计划已完成，并保存到 `doc/specs/<需求目录名>/tasks.md`。接下来有两种执行方式：

1. Subagent-Driven（推荐）: 我为每个任务分派一个新的子代理，并在任务之间做评审，迭代更快
2. Inline Execution：我在当前会话里用 executing-plans 执行，按批次推进并设置检查点

无论选择哪种执行方式，代码完成后都不会直接 `git commit`；会先做验证、评审，并确认 `/doc/feat/feat_xxxx.md` 已补齐（如适用）。

请选择一种方式。
```

**如果用户选 Subagent-Driven：**

- 必须使用 `superpowers:subagent-driven-development`
- 每个任务一个新子代理，并带两阶段评审

**如果用户选 Inline Execution：**

- 必须使用 `superpowers:executing-plans`
- 按批次执行，并在检查点回顾
