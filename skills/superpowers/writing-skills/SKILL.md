# 编写 Skills

## 概览

**写 skill，本质上就是把 TDD 用在流程文档上。**

个人 skills 一般放在代理自己的 skill 目录中。对 Codex 来说，常见位置是 `~/.agents/skills/`。

写 skill 的正确方式不是“我把经验记下来”，而是：

1. 先设计压力场景（相当于测试用例）
2. 在没有这个 skill 的情况下观察代理如何失败（RED）
3. 再写 skill 文档（GREEN）
4. 验证代理是否开始稳定遵守（GREEN）
5. 再堵漏洞、补规则、继续复测（REFACTOR）

**核心原则：**如果你没亲眼看到代理在没有 skill 时失败，你就不知道这份 skill 到底教会了什么。

**必备前置：**使用本 skill 前，必须理解 `superpowers:test-driven-development`。本 skill 只是把 TDD 的 RED-GREEN-REFACTOR 循环迁移到文档设计上。

## 什么是 Skill

Skill 是一份关于**已被证明有效**的方法、模式、工具或参考资料的说明文档，目的是让未来的代理能快速发现并正确应用它。

**Skill 是：**

- 可复用的方法
- 稳定的模式
- 工具使用说明
- 参考指南

**Skill 不是：**

- 某次问题解决过程的流水账
- 单次项目经验总结
- 只对一个仓库有用的局部约定

## TDD 与技能编写的映射关系

| TDD 概念 | Skill 编写中的对应物 |
|----------|----------------------|
| 测试用例 | 压力场景 / 子代理验证场景 |
| 生产代码 | `SKILL.md` 文档 |
| RED | 没有 skill 时代理违反规则 |
| GREEN | 有了 skill 后代理开始遵守 |
| REFACTOR | 找新漏洞、补规则、重新验证 |
| 先写测试 | 先跑基线场景，再写 skill |
| 看失败 | 记录代理到底是怎么自我合理化的 |
| 最小实现 | 只补足足以修正错误行为的规则 |
| 看通过 | 验证代理在同场景下已能遵守 |

写 skill 的全过程都应遵循 RED-GREEN-REFACTOR。

## 何时值得写一个 Skill

**适合写 skill 的情况：**

- 某个技巧对你来说并不是显而易见的
- 你未来在多个项目里还会重复用到它
- 它描述的是普适模式，而不是项目专有约定
- 它值得被别人复用

**不适合写 skill 的情况：**

- 一次性解法
- 社区里已有清晰标准答案的常规知识
- 项目专有约定（应写进 `AGENTS.md`）
- 可直接通过 lint、校验器、脚本强制执行的机械约束

## 技能类型

### 技巧型（Technique）

清晰步骤、可直接执行的方法，例如条件等待、根因追踪。

### 模式型（Pattern）

用于指导思考和拆解问题的思维模式，例如用标志位替代过深嵌套。

### 参考型（Reference）

API 文档、语法说明、工具说明等。

## 目录结构

```text
skills/
  skill-name/
    SKILL.md              # 主文档，必需
    supporting-file.*     # 必要时再拆附属文件
```

使用**扁平命名空间**，便于未来检索。

### 什么应该拆成独立文件

以下内容适合拆出去：

1. **重型参考资料**：通常 100 行以上，例如 API 说明、语法清单
2. **可复用工具**：脚本、模板、辅助文件

### 什么应该留在 `SKILL.md`

- 原则和概念
- 不超过 50 行的代码模式
- 其他大多数正文内容

## `SKILL.md` 推荐结构

### 文档头部元数据

YAML frontmatter 至少应包含两个字段：

- `name`
- `description`

约束：

- 总长度不超过 1024 字符
- `name` 只用字母、数字和连字符
- `description` 只描述“何时使用”，不要写“怎么做”

**推荐写法：**

- 用第三人称或中性陈述
- 聚焦触发条件、症状、上下文
- 尽量简洁，最好控制在 500 字符内

```markdown
---
name: skill-name-with-hyphens
description: 在 [具体触发条件与症状] 时使用
---
```

### 正文结构

```markdown
# 技能名称

## 概览
这是什么？核心原则是什么？用 1-2 句话说清。

## 何时使用
如有必要，可附一个小型流程图。
用清晰条目列出症状、适用场景和不适用场景。

## 核心模式
如果是技巧型或模式型 skill，可用前后对比代码展示。

## 快速参考
适合用表格或条目做扫描式阅读。

## 实现 / 细节
简单模式直接内联，重型资料外链到附属文件。

## 常见错误
写“会错在哪、怎么修正”。

## 实际效果（可选）
给出真实收益或案例。
```

## 技能检索优化（SSO）

未来的代理能不能发现你的 skill，首先取决于它能不能在检索阶段匹配到。

### 1. `description` 字段要“只写触发条件”

这是最容易写错的地方。

`description` 只应该回答：

**“什么时候我应该加载这个 skill？”**

而不应该回答：

**“这个 skill 里面大概怎么做。”**

#### 错误示例

```yaml
description: 在执行计划时使用，会按任务分派子代理并在任务之间做代码评审
description: 在进行 TDD 时使用，先写测试再失败再修代码
```

这类描述把流程摘要提前塞进 frontmatter，会导致代理直接拿描述字段当“简版规则”，反而不去读 skill 正文。

#### 正确示例

```yaml
description: 在当前会话中执行由多个独立任务组成的实现计划时使用
description: 在实现任何功能或修复缺陷、且尚未开始编写实现代码之前使用
```

#### 写法要求

- 写触发条件、症状、上下文
- 描述问题，而不是复述工作流
- 除非 skill 本身就是技术栈专用，否则尽量技术无关
- 如果确实是技术栈专用 skill，就明确写出技术条件

#### 更多示例

```yaml
# 差：太抽象
description: 用于异步测试

# 差：第一人称
description: 当异步测试不稳定时，我可以帮你

# 差：把触发条件写得过窄、过实现细节
description: 在测试依赖 setTimeout/sleep 且不稳定时使用

# 好：聚焦问题本身
description: 在测试存在竞态、时序依赖，或通过结果不稳定时使用

# 好：明确技术条件
description: 在使用 React Router 处理认证重定向时使用
```

### 2. 关键词覆盖

站在未来代理会怎么搜的角度去写。

应覆盖：

- 常见错误信息，例如 `"Hook timed out"`、`"ENOTEMPTY"`
- 症状，例如 `flaky`、`hang`、`zombie`
- 同义词，例如 `timeout/hang/freeze`
- 工具名、命令名、库名、文件类型

### 3. 名称要便于检索

尽量使用**动词优先、主动语态**：

- 好：`creating-skills`
- 差：`skill-creation`

- 好：`condition-based-waiting`
- 差：`async-test-helpers`

### 4. Token 效率

部分基础 skill 会在大量会话中频繁出现，所以每个 token 都很宝贵。

#### 建议长度

- 入门型 / 高频流程型 skill：尽量控制在 150-200 词级别的密度
- 其他 skill：一般控制在 500 词内会更稳

#### 降低 token 的方法

**把细节搬到工具帮助里：**

```text
差：在 SKILL.md 中穷举所有命令参数
好：只说明“支持多个过滤器，具体运行 --help 查看”
```

**用交叉引用代替重复说明：**

```text
差：在多个 skill 中反复重复同一段工作流
好：在当前 skill 中引用另一个专门 skill
```

**压缩示例：**

```text
差：用很长的对话把一个意思讲三遍
好：保留最小示例，够说明模式即可
```

**删除重复信息：**

- 不要重复别的 skill 已明确说明的内容
- 不要解释命令字面上就能看懂的东西
- 不要给多个本质一样的例子

## 写 Skill 的实际流程

### 红：先写失败场景

先设计“如果没有这份 skill，代理会怎么犯错”的场景。

记录：

- 它在什么情境下错了
- 它具体做了什么错误动作
- 它用了什么借口为自己开脱

### 绿：写最小可行技能

不要试图一次写成百科全书。  
先只解决在 RED 阶段看到的那类错误。

### 重构：堵漏洞

再次运行同样场景，看看代理有没有换一种说法继续犯同样的错。

常见现象：

- 表面遵守，实则钻规则空子
- 只遵守了一半
- 学会了措辞，但没学会动作

看到这些，就继续补规则，再复测。

## 如何验证 Skill

验证 skill 的目标，不是证明“文档写得挺像回事”，而是证明“代理真的开始按这个 skill 行动”。

### 验证时至少要回答这些问题

1. 没有 skill 时，代理会怎么失败？
2. 有了 skill 后，是否在同样场景下表现不同？
3. 它是因为读懂 skill 才成功，还是只是凑巧？
4. 是否还存在明显可钻的空子？

### 推荐验证方式

- 用子代理跑压力场景
- 同一场景在 skill 前后都跑一遍
- 记录代理的原话和动作
- 针对其借口补规则

如果只看你自己“觉得这份文档不错”，那不算验证。

## 编写 Skill 时的常见错误

| 错误 | 为什么错 | 修正方式 |
|------|----------|----------|
| 把 skill 写成故事 | 未来代理无法快速检索和执行 | 改成模式、规则、步骤 |
| `description` 写成流程摘要 | 代理可能不看正文 | 只保留触发条件 |
| 没做 RED 就开始写 | 不知道 skill 真正要教什么 | 先跑失败场景 |
| 内容过长却不拆文件 | 高频加载会浪费大量 token | 把重型参考拆出去 |
| 一次写太满 | 容易虚胖、难验证 | 从最小可行规则开始 |
| 只写“要做什么”不写“何时停下” | 代理容易硬着头皮乱做 | 增加停止条件和红旗 |

## 现实判断

好 skill 的标准不是“文笔好”，而是：

- 未来代理能发现它
- 发现后会真正去读
- 读完后会改变行为
- 面对压力场景时仍能守住规则

如果做不到这四点，它大概率就还没写好。
