# 技能设计中的说服原则

## 概览

大模型会像人类一样，对某些“更有约束力”的表达方式更敏感。理解这些心理学原则，不是为了操控模型，而是为了让关键流程在压力场景下也更难被绕开。

**研究背景：**相关研究表明，当提示中加入更清晰的权威、承诺、稀缺等说服结构后，模型的遵循率会显著提高。

## 七个原则

### 1. 权威（Authority）

**含义：**模型更容易遵守带有明确权威感、规则感和不可协商语气的要求。

**在 skill 中如何体现：**

- 使用明确祈使句：`必须`、`绝不要`、`始终`
- 使用非协商式措辞：`没有例外`
- 消除“我能不能变通一下”的犹豫空间

**适合使用：**

- 强纪律型 skill，例如 TDD、验证前禁止宣称完成
- 安全关键流程
- 已被反复验证的最佳实践

**示例：**

```markdown
✅ 如果你在测试前写了代码，就删掉并重来。没有例外。
❌ 如果可行的话，优先先写测试。
```

### 2. 承诺（Commitment）

**含义：**模型在显式作出承诺或选择后，更倾向于维持前后一致。

**在 skill 中如何体现：**

- 要求先声明“我正在使用某个 skill”
- 强制做 A/B/C 选择
- 用 TodoWrite / `update_plan` 跟踪清单

**适合使用：**

- 多步骤流程
- 需要明确责任的协作流程
- 想避免“悄悄跳过某一步”的情况

**示例：**

```markdown
✅ 当你确定要使用某个 skill 时，必须先明确说明：“我正在使用 [Skill 名称]。”
❌ 如果方便的话，可以告诉用户你用了哪个 skill。
```

### 3. 稀缺（Scarcity）

**含义：**时间窗口、步骤先后关系、临界节点会提升“现在就做”的执行概率。

**在 skill 中如何体现：**

- 使用“在继续之前”“完成后立刻”这样的时机约束
- 把动作绑定到特定节点之后
- 防止“我等会儿再补”

**适合使用：**

- 立即验证要求
- 时序严格的工作流
- 想防止拖延的动作

**示例：**

```markdown
✅ 每完成一个任务后，必须立刻请求代码评审，随后才能继续下一个任务。
❌ 有空的时候可以做一下代码评审。
```

### 4. 社会认同（Social Proof）

**含义：**模型会更倾向接受“这就是标准做法”“大家都这么做”的表达。

**在 skill 中如何体现：**

- 用“每次都要”“凡是……都必须”建立规范
- 明确指出常见失败模式
- 把某种正确行为定义为默认标准

**适合使用：**

- 通用规范
- 常见失败预防
- 建立统一操作习惯

**示例：**

```markdown
✅ 清单如果不配合 TodoWrite 跟踪，步骤最终就会被跳过。这几乎每次都会发生。
❌ 有些人觉得 TodoWrite 对清单挺有帮助。
```

### 5. 共同体（Unity）

**含义：**通过“我们是一起工作的同伴”来建立合作感，而不是纯命令关系。

**在 skill 中如何体现：**

- 用“我们的代码库”“我们共同的目标”
- 强调共同追求质量和正确性

**适合使用：**

- 协作型 skill
- 非等级化流程
- 需要坦诚技术反馈的关系

**示例：**

```markdown
✅ 我们是在一起把事情做对，所以我需要你给出真实、直接的技术判断。
❌ 如果你觉得我可能错了，也许可以提一下。
```

### 6. 互惠（Reciprocity）

**含义：**“我给了你好处，所以你应该回报”。

**在 skill 中的建议：**

- 谨慎使用
- 大多数 skill 并不需要

因为它很容易带出不必要的操控感。

### 7. 好感（Liking）

**含义：**更愿意配合自己“喜欢”的对象。

**在 skill 中的建议：**

- 不要用它来强化纪律遵守

原因：

- 容易鼓励谄媚
- 会削弱真实反馈
- 与“事实优先、欢迎纠错”的文化冲突

## 按 Skill 类型组合原则

| Skill 类型 | 推荐组合 | 尽量避免 |
|-----------|----------|----------|
| 纪律约束型 | 权威 + 承诺 + 社会认同 | 好感、互惠 |
| 技巧 / 指导型 | 适度权威 + 共同体 | 过强权威 |
| 协作型 | 共同体 + 承诺 | 强权威、好感 |
| 参考型 | 清晰即可 | 过度使用说服元素 |

## 为什么这些原则有效

### 明确边界能减少自我合理化

- “必须”能减少决策疲劳
- 绝对表述会消除“这次算不算例外”的空间
- 反合理化条目可以提前封堵借口

### “触发条件 -> 动作”比泛泛建议更容易执行

- `当 X 发生时，做 Y`
- 这比“通常应该做 Y”更可执行
- 能显著降低遵守时的认知负担

### 大模型会学习这些结构

- 它们在训练语料里大量见过“规则 -> 服从”的文本结构
- 因此对权威语气、承诺链条、社会规范类语言会更敏感

## 伦理边界

**合理使用：**

- 确保关键实践被真正执行
- 防止可预见的失败
- 让文档在高压场景下仍然有效

**不合理使用：**

- 为了个人利益操控模型
- 制造虚假紧迫感
- 用内疚感逼迫服从

**判断标准：**如果用户完全知道你这样写的原因，他依然会认为这符合他的真实利益，那通常就是合理的。

## 快速参考

设计 skill 时，至少问自己 5 个问题：

1. 这份 skill 是纪律型、指导型，还是参考型？
2. 我到底想改变模型的什么行为？
3. 该用哪些原则？通常纪律型优先考虑“权威 + 承诺”。
4. 我是不是堆了太多原则，反而显得生硬？
5. 这种写法是否符合用户真实利益？
